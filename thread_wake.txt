● 這個包下有兩個類，是一個用 sun.misc.Unsafe 手動實現
  CAS（Compare-And-Swap）計數器的示例。下面梳理本質邏輯和原理，再逐一分析。

  ---
  核心原理

  CAS 是一種**無鎖（lock-free）**的原子操作，語義為：

  ▎ 「如果當前值 == 預期的舊值，就把它更新為新值；否則什麼都不做，返回失敗。」

  JDK 的 AtomicInteger
  底層就是這麼實現的，這裡的代碼等於手動復刻了它的核心邏輯。

  靜態初始化塊

  Field f = Unsafe.class.getDeclaredField("theUnsafe");
  f.setAccessible(true);
  UNSAFE = (Unsafe) f.get(null);
  VALUE_OFFSET = UNSAFE.objectFieldOffset(
      MyCASCounter.class.getDeclaredField("value"));

  - Unsafe 的構造函數是私有的，只能通過反射拿到內部的 theUnsafe 單例。
  - objectFieldOffset 計算出 value 字段相對於對象起始地址的偏移量，後續 CAS
  操作直接按這個偏移去讀寫內存。

  incrementAndGet() — 自旋 CAS

  public int incrementAndGet() {
      for (;;) {                    // 無限循環（自旋）
          int old = value;          // 1. 讀取當前值
          int next = old + 1;       // 2. 計算新值
          if (UNSAFE.compareAndSwapInt(this, VALUE_OFFSET, old, next)) {
              return next;          // 3. CAS 成功，返回
          }
          // CAS 失敗說明有其他線程搶先修改了，重新來一輪
      }
  }

  這就是典型的 自旋 + CAS 模式：
  1. 讀舊值
  2. 算新值
  3. 用 CAS 嘗試寫入，失敗就重試

  與加鎖相比，沒有線程掛起/喚醒的開銷，在競爭不激烈時性能更好。

● 這是個非常好的問題，先把本質理清楚。

  答案：是你說的第二種

  代碼的真實執行順序就是：先算好新值，再回頭驗證舊值有沒有被改過。如果被改了，本
  次計算作廢，重來。

  回到代碼：

  for (;;) {
      int old = value;           // ① 拍一張「快照」
      int next = old + 1;        // ② 基於快照，先把結果算好
      if (UNSAFE.compareAndSwapInt(this, VALUE_OFFSET, old, next)) {
          return next;           // ③ 驗證：內存值還是 old 嗎？是→寫入 next
      }
      // 不是 → 說明別人改過了，① ② 的工作作廢，重新循環
  }

  為什麼不能「先比較，再計算」？

  你設想的模型是：

  ▎ 先比較 → 確認沒人改 → 再計算 → 再寫入

  問題在於：比較和寫入之間有時間差。在你「確認沒人改」之後、「寫入」之前，別的線
  程完全可能插進來改掉值。這樣你的寫入就覆蓋了別人的結果，數據就錯了。

  這個模型要正確，必須加鎖把「比較→計算→寫入」整段保護起來，那就不是無鎖了。

  CAS 的精髓：把「比較」和「寫入」壓成一個原子操作

  compareAndSwapInt(obj, offset, expected, newValue) 這條指令在 CPU
  層面做的事情是：

  ▎ 在同一個不可分割的瞬間：讀內存、比較、寫入，三合一。

  所以正確的流程是：

  讀快照 → 先算好結果 → 用一條原子指令「比較+寫入」
                          ↑
                     這一步不可被打斷

  計算可以提前做，因為計算本身不改內存，不影響任何人。真正關鍵的只有最後那一步原
  子操作——它同時完成了驗證和寫入，中間不可能被插隊。

  用生活類比

  ▎ 你在共享文檔上改一行字：
  ▎ 1. 看一眼當前內容（拍快照）
  ▎ 2. 在本地想好要改成什麼（先計算）
  ▎ 3. 提交時系統檢查：「這行還是你看到的那個版本嗎？」
  ▎   - 是 → 提交成功
  ▎   - 不是 → 「有人搶先改了，請基於最新版重新編輯」

  這就是樂觀並發控制（Optimistic Concurrency）——先樂觀地算，最後一刻再驗證。

  重點觀察

  看「CAS失敗」那幾行，它清楚展示了你問的流程：

  1. Thread-1 拍了快照 old=0，算好了 next=1
  2. 但提交瞬間發現內存已經被 Thread-0 改成 1 了
  3. 計算作廢，重新讀快照 old=1，重新算 next=2，再試一次

> &#x20;                                                                        
CAS線程失敗了會被安置在哪裏？它又如何什麼時機會做什麼操作後會獲取cpu的時間片    

● 這是個關鍵問題。答案可能跟你想的不一樣：

  CAS 失敗後，線程哪裡都沒去

  回到代碼的本質：

  for (;;) {           // ← 這是一個普通的死循環
      int old = value;
      int next = old + 1;
      if (UNSAFE.compareAndSwapInt(...)) {
          return next;
      }
      // 失敗了？沒有任何 wait/sleep/park
      // 直接跳回 for 的開頭，馬上重來
  }

  CAS 失敗不是一個「異常事件」，它只是一個 if 判斷返回了 false。 線程照常持有
  CPU 時間片，繼續執行下一條指令——也就是回到循環頭部再試。

  跟 synchronized 的根本區別

  這就是「自旋」和「阻塞」的本質差異：

  ┌──────────┬──────────────────────────────────┬───────────────────────────┐
  │          │       synchronized / Lock        │         CAS 自旋          │
  ├──────────┼──────────────────────────────────┼───────────────────────────┤
  │ 搶不到時 │ 線程被 park（掛起），從運行態 →  │ 什麼都不做，繼續佔著 CPU  │
  │          │ 等待態，移出 CPU                 │ 空轉                      │
  ├──────────┼──────────────────────────────────┼───────────────────────────┤
  │ 存放位置 │ 進入鎖的等待隊列（如 AQS 的 CLH  │ 沒有隊列，線程還在 CPU 上 │
  │          │ 隊列）                           │                           │
  ├──────────┼──────────────────────────────────┼───────────────────────────┤
  │ 如何恢復 │ 持鎖線程釋放鎖 → OS 喚醒 →       │ 不需要恢復，因為從未離開  │
  │          │ 重新調度 → 上下文切換            │                           │
  ├──────────┼──────────────────────────────────┼───────────────────────────┤
  │ 代價     │ 上下文切換（微秒級）             │ 空轉浪費 CPU 週期         │
  └──────────┴──────────────────────────────────┴───────────────────────────┘

  所以你問的「被安置在哪裡」——答案是沒有被安置，它一直在跑。

  那 CPU 時間片用完了怎麼辦？

  線程的時間片是由 OS 調度器 管理的，跟 CAS 無關。流程是：

  線程 A 正在自旋
      ↓
  時間片耗盡（通常 1~10ms）
      ↓
  OS 強制搶佔，切換到其他線程
      ↓
  輪到 A 時，OS 重新分配時間片
      ↓
  A 從上次 for 循環的位置繼續執行（不是從頭開始）
      ↓
  繼續自旋...

  線程不知道自己被搶佔過，恢復後就是繼續跑那個 for
  循環，跟普通代碼沒有任何區別。

  一句話總結

  ▎ CAS 自旋的線程永遠處於「運行態」或「就緒態」，從不進入「等待態」。它不靠任何
  人喚醒，只靠 OS 正常的時間片輪轉就能繼續執行。

  這也是為什麼高競爭場景下自旋不好——10 個線程搶一個值，9 個在空轉浪費
  CPU，不如讓它們掛起去睡覺。這就是 LongAdder 用分段思路解決的問題。

> 我是說在OS把cpu的時間片給其它線程的時候，這個線程在哪幹什麼？                 

● 明白了，你問的是：時間片被搶走後，這個線程在哪、處於什麼狀態。

  它在 OS 的「就緒隊列（Ready Queue）」裡排隊，什麼都不幹

  整個過程：

  線程 A 正在 CPU 上跑 for 循環          ← 運行態（Running）
           ↓
      時間片耗盡，OS 觸發時鐘中斷
           ↓
  OS 把 A 的現場（寄存器、程序計數器等）     ← 保存上下文
      存到 A 的線程控制塊（TCB）裡
           ↓
  A 被放進就緒隊列，狀態變為              ← 就緒態（Ready）
      「我隨時能跑，但沒有 CPU 給我」
           ↓
  OS 從就緒隊列裡挑另一個線程上 CPU
           ↓
      ... A 在隊列裡等 ...               ← 純等待，不消耗 CPU
           ↓
      輪到 A 了，OS 恢復它的上下文
           ↓
  A 回到 CPU，從 for 循環中斷的地方繼續跑  ← 又變回運行態

  跟「等待態」的區別

  ┌──────────────┬───────────────────────┬──────────────────────────────────┐
  │              │    就緒態（Ready）    │        等待態（Waiting）         │
  ├──────────────┼───────────────────────┼──────────────────────────────────┤
  │ 為什麼不在跑 │ CPU                   │ 我在等一個條件（鎖釋放、IO       │
  │              │ 被別人佔了，輪不到我  │ 完成等）                         │
  ├──────────────┼───────────────────────┼──────────────────────────────────┤
  │ 能不能立刻跑 │ 能，給我 CPU 就行     │ 不能，條件沒滿足，給 CPU 也沒用  │
  ├──────────────┼───────────────────────┼──────────────────────────────────┤
  │ 誰讓它恢復   │ OS 調度器按時間片輪轉 │ 必須有人顯式喚醒（notify /       │
  │              │                       │ unpark / IO 回調）               │
  ├──────────────┼───────────────────────┼──────────────────────────────────┤
  │ CAS 自旋線程 │ 在這裡                │ 永遠不會到這裡                   │
  └──────────────┴───────────────────────┴──────────────────────────────────┘

  所以回答你的問題

  ▎ 這個線程在哪？

  在 OS 內核維護的就緒隊列裡。

  ▎ 在幹什麼？

  什麼都沒幹。它的整個執行狀態（跑到 for 循環的哪一行、局部變量 old 和 next
  的值）都被凍結在 TCB 裡，等 OS 重新把 CPU 分給它，原樣恢復繼續跑。

> 就緒隊列裏還有一個等待隊列？這個隊列是幹嘛的？和就緒隊列的定位和功能分別是什  
麼？                                                                            

● 不是「就緒隊列裡面有一個等待隊列」，它們是 OS
  裡並列的兩個隊列，服務於不同目的。

  線程的三個核心狀態和兩個隊列

                           獲得 CPU 時間片
      ┌──────────────┐  ──────────────→  ┌──────────────┐
      │  就緒隊列     │                   │   運行態      │
      │  (Ready Queue)│  ←──────────────  │  (Running)   │
      └──────────────┘   時間片耗盡        └──────────────┘
            ↑                                   │
            │                                   │ 等待鎖 / IO / sleep
            │ 條件滿足，被喚醒                     ↓
      ┌──────────────┐                    ┌──────────────┐
      │              │  ←──────────────   │   等待隊列    │
      │              │    不會直接回到      │ (Wait Queue) │
      │              │    Running！        └──────────────┘
      └──────────────┘

  兩個隊列的對比

  ┌────────────┬───────────────────────┬─────────────────────────────────────┐
  │            │   就緒隊列（Ready     │       等待隊列（Wait Queue）        │
  │            │        Queue）        │                                     │
  ├────────────┼───────────────────────┼─────────────────────────────────────┤
  │ 裡面是誰   │ 萬事俱備，只缺 CPU    │ 在等某個條件的線程（鎖、IO、sleep） │
  │            │ 的線程                │                                     │
  ├────────────┼───────────────────────┼─────────────────────────────────────┤
  │ 為什麼進來 │ 時間片耗盡 / 剛被喚醒 │ 主動讓出：wait()、park()、IO 阻塞   │
  ├────────────┼───────────────────────┼─────────────────────────────────────┤
  │ 怎麼出去   │ OS 調度器分配時間片 → │ 條件滿足被喚醒 → 先進就緒隊列排隊   │
  │            │  直接上 CPU           │                                     │
  ├────────────┼───────────────────────┼─────────────────────────────────────┤
  │ 關鍵區別   │ 隨時能跑              │ 給 CPU 也沒用，條件沒滿足跑不了     │
  └────────────┴───────────────────────┴─────────────────────────────────────┘

  核心要點：等待隊列出來後不能直接跑

  這是最容易搞混的地方：

  線程 B 持有鎖，線程 A 搶不到
      ↓
  A 進入等待隊列，睡了
      ↓
  B 釋放鎖，喚醒 A
      ↓
  A 出等待隊列 → 進就緒隊列排隊      ← 注意：不是直接跑！
      ↓
  OS 輪到 A → A 上 CPU 繼續執行

  被喚醒 ≠ 立刻執行，只是獲得了排隊的資格。

  回到 CAS 自旋

  CAS 自旋的線程只在「運行態」和「就緒態」之間切換，永遠不進等待隊列。這就是它和
   synchronized 的根本區別——synchronized 搶不到鎖會把線程丟進等待隊列去睡覺，而
  CAS 寧可空轉也不睡。

> 那你再講講synchronized的原理                                                  

● 先抓本質

  synchronized 的核心是Monitor（監視器/管程）。每個 Java
  對象都能充當一把鎖，背後就是一個 Monitor。

  ---
  一、對象頭裡藏著鎖的狀態

  每個 Java 對象在內存中的佈局：

  ┌─────────────────────────────────────┐
  │  對象頭 (Object Header)              │
  │  ┌─────────────────────────────────┐ │
  │  │ Mark Word (64bit)               │ │  ← 鎖的信息全在這裡
  │  │ [鎖標誌位 | 線程ID | hashcode...] │ │
  │  └─────────────────────────────────┘ │
  │  ┌─────────────────────────────────┐ │
  │  │ 類型指針 (Klass Pointer)         │ │
  │  └─────────────────────────────────┘ │
  ├─────────────────────────────────────┤
  │  實例數據                            │
  └─────────────────────────────────────┘

  Mark Word 的最後 2~3 位是鎖標誌位，標識當前是什麼鎖狀態。

  ---
  二、鎖升級：從輕到重

  JDK 6 之後，synchronized 不是一上來就重量級，而是逐步升級：

  無鎖 → 偏向鎖 → 輕量級鎖 → 重量級鎖
                  （只升不降）

  1. 偏向鎖（Biased Locking）

  場景：只有一個線程反覆進入同步塊，根本沒有競爭。

  線程 A 第一次進入 synchronized 塊
      ↓
  把 A 的線程 ID 寫入 Mark Word
      ↓
  A 之後再進入同一把鎖，發現 Mark Word 裡就是自己的 ID
      ↓
  直接進入，不做任何同步操作（幾乎零成本）

  ▎ 本質：記住上次是誰拿的鎖，如果還是它，直接放行。

  2. 輕量級鎖（Lightweight Lock）

  場景：有少量線程交替進入，但時間上沒有重疊（沒有真正的競爭）。

  線程 A 持有偏向鎖，線程 B 也來了
      ↓
  偏向鎖撤銷，升級為輕量級鎖
      ↓
  線程在自己的棧幀中創建 Lock Record
      ↓
  用 CAS 把 Mark Word 替換為指向 Lock Record 的指針
      ↓
  CAS 成功 → 拿到鎖
  CAS 失敗 → 自旋幾次重試
  自旋還是失敗 → 升級為重量級鎖

  ▎ 本質：用 CAS + 短暫自旋代替 OS 級別的阻塞，賭對方很快就會釋放。

  3. 重量級鎖（Heavyweight Lock）

  場景：真正的多線程激烈競爭。

  自旋多次還搶不到
      ↓
  升級為重量級鎖，Mark Word 指向一個 OS 的 Monitor 對象
      ↓
  搶不到鎖的線程被 OS 掛起（park），進入等待隊列

  ▎ 本質：回到傳統的 OS 互斥鎖，搶不到就睡覺，靠喚醒機制恢復。

  ---
  三、Monitor 的內部結構

  重量級鎖背後就是這個 Monitor：

  ┌─────────────────────────────────────┐
  │            Monitor                   │
  │                                      │
  │   Owner:  當前持有鎖的線程            │
  │                                      │
  │   EntryList (入口隊列/阻塞隊列):      │
  │   ┌───┐ ┌───┐ ┌───┐                 │
  │   │ B │→│ C │→│ D │  搶鎖失敗，在這排隊│
  │   └───┘ └───┘ └───┘                 │
  │                                      │
  │   WaitSet (等待集合):                 │
  │   ┌───┐ ┌───┐                       │
  │   │ E │→│ F │  調了 wait()，在這等通知 │
  │   └───┘ └───┘                       │
  │                                      │
  └─────────────────────────────────────┘

  EntryList vs WaitSet

  ┌────────────┬─────────────────────────┬───────────────────────────┐
  │            │        EntryList        │          WaitSet          │
  ├────────────┼─────────────────────────┼───────────────────────────┤
  │ 怎麼進來   │ synchronized 搶鎖失敗   │ 持有鎖的線程主動調 wait() │
  ├────────────┼─────────────────────────┼───────────────────────────┤
  │ 在等什麼   │ 等 Owner 釋放鎖         │ 等別人調 notify()         │
  ├────────────┼─────────────────────────┼───────────────────────────┤
  │ 出去後去哪 │ 直接競爭鎖 → 成為 Owner │ 先進 EntryList → 再競爭鎖 │
  └────────────┴─────────────────────────┴───────────────────────────┘

  完整的流程：

  線程 A 持有鎖，在同步塊裡執行
  線程 B、C、D 來搶鎖 → 失敗 → 進 EntryList 掛起

  情況一：A 執行完 synchronized 塊，釋放鎖
      → 從 EntryList 喚醒一個線程去搶鎖

  情況二：A 在塊裡調了 wait()
      → A 釋放鎖，自己進 WaitSet 睡覺
      → EntryList 裡的線程被喚醒去搶鎖
      → 未來某個線程調 notify()
      → A 從 WaitSet 出來，進 EntryList 重新排隊搶鎖

  ---
  四、和 CAS 自旋的對比（串起來）

                    競爭程度
      低 ──────────────────────────→ 高

     偏向鎖        輕量級鎖          重量級鎖
     (無競爭)    (CAS + 自旋)     (OS 掛起/喚醒)
     零成本       空轉消耗 CPU      上下文切換開銷

                      ↑
                你之前看的 CAS 自旋
                就是在這個階段發生的

  所以 synchronized 並不慢——它內部已經包含了 CAS
  自旋的優化。只有競爭真正激烈到自旋解決不了時，才會退化到重量級鎖讓線程去睡覺。

> 競爭失敗不是先到cxq，再到entrylist嗎？                                        

● 你说得对，我之前简化了。完整的流程确实是先进 cxq，再转移到 EntryList。

  完整的三个队列

  ┌──────────────────────────────────────────┐
  │              Monitor                      │
  │                                           │
  │   Owner:  当前持有锁的线程                  │
  │                                           │
  │   cxq (Contention Queue):                 │
  │   ┌───┐ ┌───┐ ┌───┐                      │
  │   │ D │→│ C │→│ B │  刚抢锁失败，先到这里    │
  │   └───┘ └───┘ └───┘                      │
  │                                           │
  │   EntryList:                              │
  │   ┌───┐ ┌───┐                            │
  │   │ X │→│ Y │  从 cxq 转移过来，等待竞争锁  │
  │   └───┘ └───┘                            │
  │                                           │
  │   WaitSet:                                │
  │   ┌───┐ ┌───┐                            │
  │   │ E │→│ F │  调了 wait()，等 notify()    │
  │   └───┘ └───┘                            │
  │                                           │
  └──────────────────────────────────────────┘

  为什么要分 cxq 和 EntryList 两个队列？

  职责不同：

  ┌──────────┬──────────────────────────────┬───────────────────────────────┐
  │          │             cxq              │           EntryList           │
  ├──────────┼──────────────────────────────┼───────────────────────────────┤
  │ 是什么   │ 竞争入口，一个栈（LIFO）     │ 候选队列，链表                │
  ├──────────┼──────────────────────────────┼───────────────────────────────┤
  │ 谁往里放 │ 抢锁失败的线程自己用 CAS     │ Owner 释放锁时批量从 cxq      │
  │          │ 插入                         │ 搬过来                        │
  ├──────────┼──────────────────────────────┼───────────────────────────────┤
  │ 并发压力 │ 高——多个线程同时往里塞       │ 低——只有 Owner 一个线程操作   │
  ├──────────┼──────────────────────────────┼───────────────────────────────┤
  │ 设计意图 │ 用 CAS                       │ 给 Owner                      │
  │          │ 无锁地接住高并发的入队请求   │ 一个稳定的队列来选择唤醒谁    │
  └──────────┴──────────────────────────────┴───────────────────────────────┘

  ▎ 本质：cxq 是高并发写入的缓冲区，EntryList 
  是单线程读取的工作区。分离读写，减少竞争。

  完整流程

  线程 B 抢锁失败
      ↓
  B 用 CAS 把自己插入 cxq 头部（栈顶）
      ↓
  B 被 park（挂起）
      ↓
      ...Owner A 执行完毕，释放锁...
      ↓
  A 检查 EntryList：
      ├─ 不为空 → 从 EntryList 里取一个唤醒
      └─ 为空   → 把 cxq 整个"搬"到 EntryList → 再从 EntryList 取一个唤醒
      ↓
  被唤醒的线程竞争锁 → 成为新的 Owner

  WaitSet 的线程回来时也是走 cxq

  线程 E 之前调了 wait()，在 WaitSet 里
      ↓
  某线程调 notify()
      ↓
  E 从 WaitSet 出来 → 插入 cxq（不是直接进 EntryList）
      ↓
  等 Owner 释放锁时，随 cxq 搬到 EntryList → 再竞争

  串起来看

  抢锁失败 ──→ cxq（CAS无锁入队，高并发缓冲）
                      │
            Owner 释放锁时批量搬移
                      ↓
                EntryList（稳定的候选队列）
                      │
                唤醒一个去竞争
                      ↓
                   Owner（拿到锁，执行同步块）
                      │
                调 wait()
                      ↓
                 WaitSet（等 notify）
                      │
                被 notify()
                      ↓
                    cxq ──→ 重新走一遍流程

> 再問一個問題，cpu是先把entryList裏的線程全部拿萬，在把cxq的隊列線程版過來？還 
是消耗一個，版過來一個？這個中間做成是怎樣的？                                  

● 先纠正一点：这个操作不是 CPU 做的，是当前释放锁的 Owner 
  线程在退出同步块时做的。

  答案：整个 cxq 一次性批量搬，不是一个一个搬

  来看实际过程。Owner 释放锁时（ObjectMonitor::exit）的逻辑：

  Owner 执行完同步块，准备释放锁
      ↓
  先看 EntryList 是否为空？
      │
      ├─ 不为空 → 直接从 EntryList 取一个唤醒，完事
      │           （根本不碰 cxq）
      │
      └─ 为空   → 把 cxq 整条链表一次性摘下来
                      ↓
                 放进 EntryList
                      ↓
                 从 EntryList 取一个唤醒

  为什么是批量搬而不是一个一个搬？

  因为摘 cxq 只需要一次 CAS 操作：

  // 伪代码，对应 HotSpot 源码逻辑
  for (;;) {
      Node* head = cxq;            // 拿到 cxq 链表头
      if (CAS(&cxq, head, NULL)) { // 一次 CAS 把 cxq 整个置空
          EntryList = head;        // 整条链表挂到 EntryList
          break;
      }
  }

  一次 CAS 把 cxq 指针置为 NULL，整条链就归 Owner 私有了。此后 Owner
  独占这条链，不需要任何同步就能操作 EntryList。

  如果一个一个搬，每搬一个都要做一次 CAS 跟其他正在入队的线程竞争，开销大得多。

  时间线举例

  时刻1: Owner=A, EntryList=[X,Y], cxq=[D,C,B]

      A 释放锁
      → EntryList 不为空，取 X 唤醒
      → X 竞争成功，成为 Owner
      → cxq 没动

  时刻2: Owner=X, EntryList=[Y], cxq=[D,C,B]

      X 释放锁
      → EntryList 不为空，取 Y 唤醒
      → cxq 还是没动

  时刻3: Owner=Y, EntryList=[], cxq=[D,C,B,E]  (E 新来的)

      Y 释放锁
      → EntryList 为空！
      → 一次 CAS 把 cxq 整个摘下来 → cxq 变为 []
      → EntryList = [D,C,B,E]
      → 取 D 唤醒

  时刻4: Owner=D, EntryList=[C,B,E], cxq=[]

  所以完整的节奏是

  ▎ EntryList 消耗完了，才去 cxq 
  搬一批过来。搬的时候是一次性全搬，不是用一个搬一个。

  这个设计跟消息队列的「批量拉取」思路一样——攒一批处理比逐条处理效率高得多。

> 還有一個問題，就是上一個線程cpu時間片完了，它如何決定把鎖給entrylist裏的哪個  
線程？這個entrylist是順序隊列嗎？先進先出？還是其它？                           
                                                                                
可以單獨喚醒指定的線程嗎？                                                      

● 先纠正一个前提

  不是「时间片用完了把锁给别人」。时间片用完，线程只是暂停执行，锁还握在它手里。
   只有线程执行完 synchronized 块（或调了 wait()），才会释放锁并选人唤醒。

  ---
  从 EntryList 里选谁？

  HotSpot 默认策略：取 EntryList 的头节点。

  但 EntryList 的顺序不是你想的 FIFO：

  cxq 是栈（LIFO），后来的插在头部：

  线程到达顺序:  B → C → D
  cxq 实际顺序: [D, C, B]  （D 在头部）

  Owner 释放锁时，把 cxq 整个搬到 EntryList：
  EntryList = [D, C, B]

  取头节点 → 唤醒 D（最后来的）

  所以默认行为是接近 LIFO（后进先出），不是 FIFO。 最后来排队的反而先被唤醒。

  更复杂的是：新来的线程可以插队

  Owner A 释放锁
      ↓
  唤醒 EntryList 头部的 D
      ↓
  D 还没完全醒过来（OS 恢复上下文需要时间）
      ↓
  这时线程 G 刚好执行到 synchronized，直接尝试 CAS 抢锁
      ↓
  G 抢到了！D 白醒了，继续回去排队

  ▎ synchronized 是非公平锁。 新来的线程可以跳过整个队列直接抢，不需要排队。

  ---
  可以指定唤醒某个线程吗？

  synchronized + wait/notify：不能

  synchronized (lock) {
      lock.notify();     // 唤醒 WaitSet 中的「某一个」，JVM 选，你控制不了
      lock.notifyAll();  // 唤醒 WaitSet 中的全部，但谁先拿到锁也不确定
  }

  你没有任何手段说「我要唤醒线程 B」。

  ReentrantLock + Condition：可以分组唤醒

  ReentrantLock lock = new ReentrantLock();
  Condition producerCond = lock.newCondition();  // 生产者专用
  Condition consumerCond = lock.newCondition();  // 消费者专用

  // 生产者等待
  producerCond.await();

  // 精确唤醒生产者，不影响消费者
  producerCond.signal();

  每个 Condition 有自己独立的等待队列，signal() 只唤醒该队列里的线程。虽然不能精
  确到某一个线程，但可以按业务角色分组，实现定向唤醒。

  LockSupport：可以精确到具体线程

  Thread t = new Thread(() -> {
      LockSupport.park();   // 挂起自己
      System.out.println("被唤醒了");
  });
  t.start();

  LockSupport.unpark(t);    // 精确唤醒线程 t

  这是最底层的方式，直接指定线程对象来唤醒。AQS 内部就是用它实现的。
